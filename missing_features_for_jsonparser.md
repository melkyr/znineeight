# Missing Features and Bugs found during JSON Parser "Baptism of Fire"

During the compilation attempt of the revised JSON parser example, the following critical issues, missing features, and bugs were identified in the `retrozig` bootstrap compiler.

## 1. Compiler Bugs & Stability Issues

### 1.1 Segmentation Fault in `TypeChecker::visitFnBody`
- **Status**: Critical Bug.
- **Observation**: The compiler segfaults during semantic analysis when visiting function bodies that contain complex control flow (especially switches with captures) and return path analysis.
- **Valgrind Analysis**: Indicated an invalid read of size 8 at `TypeChecker::visitFnBody` + 0x20 offset, likely due to a NULL pointer dereference when accessing `fn_symbol->symbol_type->as.function`.
- **Trigger**: Likely occurs when a function signature fails to resolve fully but the compiler proceeds to visit the body.

### 1.2 Unstable Identifier Resolution for Lifted Expressions
- **Status**: Bug.
- **Observation**: In some cases, identifiers generated by the `ControlFlowLifter` (e.g., `__tmp_if_1`) are reported as "undeclared" in subsequent passes, or symbols in `extern` calls are lost when nested within `try` or `orelse`.

## 2. Syntax & Feature Requirements (Z98 Specific)

### 2.1 Tagged Union Field Types
- **Status**: Requirement.
- **Observation**: The parser does not support "naked" tags in `union(enum)`. Every field must have an explicit type.
- **Workaround**: Use `void` for tags without payloads.
  ```zig
  // Fails
  pub const JsonValue = union(enum) { Null, ... };
  // Works
  pub const JsonValue = union(enum) { Null: void, ... };
  ```

### 2.2 No Method Support in Structs
- **Status**: Unsupported.
- **Observation**: Attempting to define `fn` inside a `struct` or `union` triggers a syntax error ("Methods are not supported in struct declarations").
- **Workaround**: Use top-level functions that take a pointer to the struct as the first argument.

### 2.3 Strict Member Access
- **Status**: Limitation.
- **Observation**: Chained member access on expressions like `obj[i].key.ptr` sometimes fails or triggers "member access only allowed on structs..." errors if the intermediate results are not explicitly typed.
- **Workaround**: Break down complex accesses into multiple assignments or use explicit `*JsonValue` pointers for recursive types.

## 3. C Interop & Type Coercion

### 3.1 String Literal Coercion
- **Status**: Limited.
- **Observation**: String literals do not always implicitly coerce to `[*]const u8` when passed to `extern` functions or assigned to pointers.
- **Requirement**: Often requires explicit `@ptrCast([*]const u8, "string")` or using the `.ptr` property of an intermediate slice.

### 3.2 Slice vs Pointer at Boundaries
- **Status**: Both Supported.
- **Observation**:
  - Many-item pointers `[*]const u8` are mapped directly to `unsigned char const*` in C89, providing standard C-string interop.
  - Slices `[]const u8` are supported at `extern` boundaries but are passed by value as a C struct (`struct { T* ptr; size_t len; }`).
  - **Verdict**: Use `[*]const u8` for compatibility with existing C libraries (like `fopen`), and `[]const u8` for Zig-aware C functions.

## 4. Parser/Lifter Ambiguities

### 4.1 While Loop Continue Expressions
- **Status**: Sensitive.
- **Observation**: The `while (cond) : (iter) statement` syntax can sometimes confuse the `ControlFlowLifter` if the body is braceless and contains other control flow.
- **Recommendation**: Use braced bodies for complex `while` loops in the bootstrap phase.
