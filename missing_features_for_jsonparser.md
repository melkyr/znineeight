# Missing Features and Bugs found during JSON Parser "Baptism of Fire"

During the compilation attempt of the revised JSON parser example, the following critical issues, missing features, and bugs were identified in the `retrozig` bootstrap compiler.

## 1. Compiler Bugs & Stability Issues

### 1.1 Segmentation Fault in `TypeChecker::visitFnBody`
- **Status**: Critical Bug.
- **Observation**: The compiler segfaults during semantic analysis when visiting function bodies that contain complex control flow (especially switches with captures) and return path analysis.
- **Failing Sample**:
  ```zig
  fn parseBoolean(p: *Parser) ParseError!JsonValue {
      if (p.input.len - p.pos >= 4 and p.input[p.pos] == 't' and p.input[p.pos+1] == 'r' and p.input[p.pos+2] == 'u' and p.input[p.pos+3] == 'e') {
          p.pos += 4;
          return JsonValue{ .Boolean = true }; // FAILS HERE if result is not assigned to a var first
      }
      // ...
  }
  ```
- **Valgrind Analysis**: Indicated an invalid read of size 8 at `TypeChecker::visitFnBody` + 0x20 offset, likely due to a NULL pointer dereference when accessing `fn_symbol->symbol_type->as.function`.
- **Trigger**: Direct returns of complex initializers (like union/struct literals) from functions returning `error!T` when preceded by complex boolean expressions.

### 1.2 Unstable Identifier Resolution for Lifted Expressions
- **Status**: Bug.
- **Observation**: Identifiers generated by the `ControlFlowLifter` (e.g., `__tmp_if_1`) are sometimes reported as "undeclared" if they are used in nested `extern` calls.

## 2. Syntax & Feature Requirements (Z98 Specific)

### 2.1 Tagged Union Field Types
- **Status**: Requirement.
- **Observation**: The parser does not support "naked" tags in `union(enum)`. Every field must have an explicit type.
- **Failing Sample**:
  ```zig
  pub const JsonValue = union(enum) {
      Null, // ERROR: Expected ':' after field name
      Boolean: bool,
  };
  ```
- **Workaround**: Use `void` for tags without payloads.
  ```zig
  pub const JsonValue = union(enum) {
      Null: void,
      Boolean: bool,
  };
  ```

### 2.2 No Method Support in Structs
- **Status**: Unsupported.
- **Observation**: Attempting to define `fn` inside a `struct` or `union` triggers a syntax error.
- **Failing Sample**:
  ```zig
  const Parser = struct {
      pos: usize,
      fn peek(self: *Parser) u8 { // ERROR: Methods are not supported
          return self.input[self.pos];
      }
  };
  ```
- **Workaround**: Use top-level functions that take a pointer to the struct as the first argument.

### 2.3 Strict Member Access
- **Status**: Limitation.
- **Observation**: Chained member access on expressions like `obj[i].key.ptr` sometimes fails.
- **Failing Sample**:
  ```zig
  __bootstrap_print(obj[i].key.ptr); // ERROR: member access only allowed on structs...
  ```
- **Workaround**: Break down complex accesses into multiple assignments.

## 3. C Interop & Type Coercion

### 3.1 String Literal Coercion
- **Status**: Limited.
- **Observation**: String literals do not always implicitly coerce to `[*]const u8`.
- **Failing Sample**:
  ```zig
  const f = fopen(path.ptr, "rb"); // ERROR: type mismatch
  ```
- **Requirement**: Often requires explicit `@ptrCast`.
  ```zig
  const rb_str: [*]const c_char = @ptrCast([*]const c_char, "rb");
  const f = fopen(@ptrCast([*]const c_char, path.ptr), rb_str);
  ```

### 3.2 Slice vs Pointer at Boundaries
- **Status**: Both Supported.
- **Observation**:
  - Many-item pointers `[*]const u8` map to `char*`.
  - Slices `[]const u8` map to C structs.
  - **Verdict**: Use `[*]const u8` for compatibility with standard C libraries.

## 4. Parser/Lifter Ambiguities

### 4.1 While Loop Continue Expressions
- **Status**: Sensitive.
- **Observation**: `while (cond) : (iter) statement` can confuse the lifter if the body is braceless.
- **Failing Sample**:
  ```zig
  while (pos < len) : (pos += 1)
      if (input[pos] == ' ') continue; // Can trigger lifter errors
  ```
- **Recommendation**: Use braced bodies for complex loops.
