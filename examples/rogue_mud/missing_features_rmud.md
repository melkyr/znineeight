# C89 Compliance Report - Rogue MUD

## Overview
This document reports on the C89 compliance of the Rogue MUD server code generated by `zig0` (Milestone 11). Tests were performed using `gcc -m32 -std=c89 -pedantic`.

## Major Findings

### 1. Tagged Union Initializers (Designated Initializers)
The compiler emits designated initializers for tagged unions in expression context (like within a loop or branch):
```c
struct zS_6eff45_EntityType __tmp_agg_8_9 = {.tag = zE_6eff45_EntityType_Tag_Player};
```
**Status**: ⚠️ Non-compliant with C89 (C90). C89 only supports positional initializers and only for the first member of a union.
**Cause**: The `C89Emitter` uses designated initializers for temporary variables generated by the `ControlFlowLifter` when a tagged union is produced as part of an expression.
**Workaround**: Use the compiler's "decomposition" strategy which assigns fields individually after declaration. This is already used for local variables in many cases, but not currently for all lifter-generated temporaries.

### 2. Duplicate 'const' Declaration Specifiers
In `ui.c`, string constants are emitted with redundant `const` qualifiers:
```c
static const unsigned char const (* zC_f270a4_ansi_reset_val)[4] = "\033[0m";
```
**Status**: ⚠️ Non-compliant with C89 (C90). While many compilers ignore redundant `const`, strict C89 (and `pedantic` mode) rejects it.
**Cause**: This happens when a Zig `const` variable is declared with a `[]const u8` type. The emitter adds `const` because the Zig variable is `const`, and another `const` because the element type is `const`.

### 3. Incompatible Pointer Types for String Initializers
String literals in `ui.c` are assigned to `unsigned char (*)[N]` pointers:
```c
static const unsigned char const (* zC_f270a4_ansi_reset_val)[4] = "\033[0m";
```
**Status**: ⚠️ Warning. C string literals are of type `char[]`, which is incompatible with `unsigned char[]` without a cast in strict modes.
**Cause**: In Z98, `u8` maps to `unsigned char`. Zig string literals are `[]const u8`, leading to this signedness mismatch during initialization.

### 4. Pointer Signedness in Slice Construction
Calls to `__make_slice_u8` often trigger signedness warnings:
```c
return __make_slice_u8(&(*zC_f270a4_ansi_reset_val)[0], 4 - 0);
```
**Status**: ⚠️ Warning. The first argument is `const unsigned char *`, but the helper expects `const char *`.
**Cause**: `zig_special_types.h` defines `__make_slice_u8` to take `const char*` to be "friendly" to string literals, but it doesn't handle pointers derived from `unsigned char` arrays without a cast.

### 5. Long Long Usage
The generated code uses `long long` (or `i64`/`u64` typedefs) for 64-bit integers in `zig_compat.h`.
**Status**: ⚠️ Non-compliant with C89 (C90), but supported by almost all 1998-era compilers (including OpenWatcom and MSVC 6.0 via `__int64`).
**Recommendation**: Accept as a necessary extension for 64-bit support.

### 6. Union Definitions inside Structs
Previously, the compiler emitted:
```c
union ErrorUnion_void { ... };
union ErrorUnion_void data;
```
Which triggered "declaration does not declare anything" warnings.
**Status**: ✅ FIXED. The compiler now combines these into `union { ... } data;` where appropriate.

### 7. Void Returning Try/Catch
Previously, `try` or `catch` returning `void` were skipped by the lifter, causing missing side effects.
**Status**: ✅ FIXED. The lifter now correctly handles void-returning control flow.

### 8. Potential Null Pointer Dereferences
The `zig0` static analyzer correctly identifies several potential null pointer dereferences in the `rogue_mud` codebase, especially around arena allocations and array indexing.
**Status**: ⚠️ Developer responsibility. These are warnings from the Zig compiler itself, not the C compiler.

## Summary for Next Tasks
To achieve full "Pedantic" C89 compliance:
- Refactor `C89Emitter` to avoid designated initializers for all aggregate temporaries.
- Deduplicate `const` qualifiers in the emitter.
- Add explicit casts when initializing `u8` pointers from string literals.
- Ensure `__make_slice_u8` calls have consistent signedness.
