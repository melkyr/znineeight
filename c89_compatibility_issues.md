# Z98 C89/C++98 Compatibility Report

This document analyzes compatibility issues in the Z98 bootstrap compiler (`zig0`) and the generated C code when targeting legacy environments like MSVC 6.0 and OpenWatcom.

## 1. C++ 98 Non-Compliant Issues (Compiler Source)

These issues affect the compilation of `zig0` itself using legacy C++ compilers like MSVC 6.0 (`cl.exe`).

### 1.1 Use of `unsigned long long`
- **Location:** `src/bootstrap/type_checker.cpp` (Line 6830).
- **Issue:** `long long` is a C++11 (and C99) feature. MSVC 6.0 does not recognize it and will report a syntax error (error C2632: 'long' followed by 'long' is illegal).
- **Example:**
  ```cpp
  (unsigned long long)node->as.integer_literal.value
  ```
- **Proposed Fix:** Cast to `u64` instead, which is correctly defined in `src/include/common.hpp`.
- **Files to Modify:** `src/bootstrap/type_checker.cpp`, method `TypeChecker::evaluateConstantExpression`.

### 1.2 Use of `<stdint.h>` or `<cstdint>`
- **Location:** `src/include/common.hpp` and `src/include/utils.hpp`.
- **Issue:** `stdint.h` and `cstddef` (specifically for `size_t` in some contexts) may not be available or fully compliant in legacy environments.
- **Status:** Currently handled via `#ifdef _MSC_VER` in `common.hpp`, but OpenWatcom might need similar treatment.
- **Proposed Fix:** Use `__STDC_VERSION__` checks to detect C89/C90 environments and provide fallbacks for fixed-width types.

### 1.3 Unused Parameters and Variables
- **Location:** Throughout `src/bootstrap/*.cpp`.
- **Issue:** While not strictly non-compliant, legacy compilers often treat these as warnings that block "warning-free" builds or trigger errors in strict modes.
- **Proposed Fix:** Use the `RETR_UNUSED(x)` macro consistently to suppress these warnings.

## 2. C89 Compliance Issues (Generated C Code)

These issues are observed in the C code generated by `zig0`.

### 2.1 Use of `long long` in Runtime Headers
- **Location:** `src/include/zig_runtime.h`.
- **Issue:** The `i64` and `u64` typedefs use `long long` for non-MSVC compilers. This triggers warnings under `-std=c89 -pedantic` in GCC and will fail on any strict C89 compiler that doesn't support 64-bit integers as an extension.
- **Proposed Fix:** Use `__int64` or other compiler-specific extensions if available, or allow the user to provide a custom 64-bit type.

### 2.2 Function Pointer to `void*` Conversions
- **Location:** Generated calls for builtins in `main.c` of `lisp_interpreter_curr`.
- **Issue:** ISO C89/C90 forbids the conversion of a function pointer to an object pointer (`void*`). This triggers a pedantic warning in GCC (`warning: ISO C forbids conversion of function pointer to object pointer type [-Wpedantic]`).
- **Proposed Fix:** In `C89Emitter`, when emitting builtin calls, use a generic function pointer type instead of `void*`.
- **Files to Modify:** `src/bootstrap/codegen.cpp`.

### 2.3 Signedness Mismatches with `__bootstrap_print`
- **Location:** Generated calls like `__bootstrap_print("\n");`.
- **Issue:** `__bootstrap_print` expects `const unsigned char*`, but C string literals are `char*`.
- **Proposed Fix:** Change runtime helpers to expect `const char*` and perform the cast to `unsigned char*` internally if necessary, or update the emitter to cast literals at the call site.
- **Files to Modify:** `src/include/zig_runtime.h` and `src/runtime/zig_runtime.c`.

### 2.4 Unused Labels
- **Location:** Generated code for loops in `mandelbrot.c`, `quicksort.c`, etc.
- **Issue:** Every loop generates a continue label (e.g., `__loop_0_continue: ;`), but if the loop doesn't contain a `continue` statement, the label remains unused, triggering warnings (`warning: label ‘__loop_0_continue’ defined but not used [-Wunused-label]`).
- **Proposed Fix:** Update `C89Emitter` to track label usage during statement emission.
- **Files to Modify:** `src/bootstrap/codegen.cpp`, methods `emitFor`, `emitWhile`, `emitBreak`, and `emitContinue`.

## 3. MSVC 6.0 and OpenWatcom Compatibility

Specific issues that block compilation on these target compilers.

### 3.1 64-bit Integer Literal Suffixes
- **Issue:** MSVC 6.0 does not support `LL` or `ULL`. It requires `i64` and `ui64`.
- **Status:** Partially implemented in `C89Emitter::emitIntegerLiteral`.
- **Proposed Fix:** Ensure all 64-bit literals consistently use the correct suffix based on the target compiler.

### 3.2 Identifier Length Limit (31 Characters)
- **Issue:** Standard C89 only guarantees that the first 31 characters of an internal identifier are significant. Some legacy linkers have even shorter limits for external symbols (e.g., 6 characters).
- **Status:** Handled by the `NameMangler`, which truncates and hashes names to 31 characters.
- **Proposed Fix:** Verification is needed for OpenWatcom to see if it requires even shorter external names.

### 3.3 The `inline` Keyword
- **Issue:** `inline` is not part of C89. MSVC 6.0 uses `__inline`.
- **Location:** `src/include/*.hpp` and generated `zig_special_types.h`.
- **Proposed Fix:** Use a macro for `inline` that resolves to `__inline` on MSVC or remains empty on strict C89 compilers.

### 3.4 Mixed Declarations and Code
- **Issue:** C89 requires all declarations to appear at the beginning of a block.
- **Status:** `C89Emitter` uses a two-pass approach to move declarations to the top.
- **Known Failure:** Temporary variables generated during expression lifting (e.g., in `if` conditions or `switch` expressions) must also be declared at the top of the enclosing block.
- **Proposed Fix:** Ensure `makeTempVarForType` is always called with `emit_decl=true` from a context that can place the declaration at the block's start.

### 3.5 Boolean Type
- **Issue:** C89 has no `bool` type.
- **Status:** `zig_runtime.h` defines `typedef int bool;`.
- **Proposed Fix:** This is generally acceptable, but ensure it doesn't conflict with any compiler-provided headers.
